VI.              Promoting Dev instance to Prod

a.     Stop the Apache daemon on both the iuatidmgmtapp01 and idmgmtapp01 servers.

b.     Delete all files and folders in the C:\Apache24 folder in the idmgmtapp01 server. (Leave the base Apache24 folder there)

c.      On the idmgmtapp01 server, map a drive to the C:\Apache24 folder on iuatidmgmtapp01.

                                                              i.      This will present itself as Z: drive (Or whatever drive letter is free) on idmgmtapp01.

d.     On the idmgmtapp01 server, copy all the files and folders from Z:\Apache24 to C:\Apache24 except the C:\Apache24\credentials folder.

                                                              i.      You are basically copying the web files from iuatidmgmtapp01 to idmgmtapp01.

                                                            ii.      If you accidently copy the C:\Apache24\credentials folder to idmgmtapp01 (or if the C:\Apache24\credentials folder was not on idmgmtapp01 to begin with), you will need to execute the following steps:

1.     Log into the idmgmtapp01 production web server using the service account srv_IDMAppUI@eversana.com.

2.     Open a PowerShell window as Administrator.

3.     In secret server, find the passwords for these three accounts:

a.     AzureAutomation@eversana.com

b.     srv_OneDriveRetention@eversana.com

c.      srv_powerbi@eversana.com

4.     In the PowerShell window, issue these three commands:

a.     AzureAutomation@eversana.com account password'| ConvertTo-SecureString AsPlainText Force | ConvertFrom-SecureString|Out-File -FilePath C:\Apache24\credentials\EncryptedAzureAutomationPassword.txt

b.     srv_OneDriveRetention@eversana.com account password'| ConvertTo-SecureString AsPlainText Force | ConvertFrom-SecureString|Out-File -FilePath C:\Apache24\credentials\EncryptedOneDriveRetentionPassword.txt

c.      srv_powerbi@eversana.com account password'| ConvertTo-SecureString AsPlainText Force | ConvertFrom-SecureString|Out-File -FilePath C:\Apache24\credentials\EncryptedPowerBiPassword.txt

5.     This will reset the encrypted passwords to that of the srv_IDMAppUI@eversana.com service accounts which runs the daemon of the web site account.

6.     Failure to do so will cause the applications to fail as they cannot be logged into using their service accounts.

e.     The next part will be updating particular syntax on the new files that were copied to idmgmtapp01. Note, all the files you update will be on idmgmtapp01.

f.       There are some specific changes when it comes to updating these files. I will cover them in line items ii and iii below. Line i is for all files.

                                                              i.      Using a file editor like Notepad++, change the syntax http://iuatidmgmtapp01 to http://idmgmtapp01 in all files that have this text in them.

                                                            ii.      In the file C:\Apache24\htdocs\webpages\ WebTitleBanner.htm on line 664, change the syntax http://iuatidmgmtapp01/css/styles.css to http://idmgmtapp01/css/styles.css .

                                                         iii.      In the file C:\Apache24\htdocs\js\functions.js, make the follow changes throughout the file:

1.     Change all instances of DevEncryptedKey to ProdEncryptedKey

2.     Change all instances of DevEmpID to ProdEmpID.

g.     Once all these changes have been made, start up Apache services on both iuatidmgmtapp01 and idmgmtapp01.

h.     Thats it! That is how you promote the dev code to production.

VII.            The Admin Portal

a.     Description

                                                              i.      The IDM Web Site provides many sources of information for various areas within HR and IDM.

                                                            ii.      It also provides applications to perform actions such as employee terminations (among others) that not everyone should have access to.

                                                         iii.      For that reason, these applications are put behind a secure area called the Admin Portal.

                                                          iv.      Users who need access to the Admin Portal should submit a request.

                                                            v.      This request can come in the form of an E-Mail or Service Now ticket.

                                                          vi.      Once approved, there is an IDM Housekeeping Web Site only accessible to Admins of the Web Site where access can be administered.

                                                       vii.      The following describes adding, removing, revoking and fixing Admin Portal access

b.     Adding user access to the Admin Portal

                                                              i.      Using your web browser, navigate to http://idmgmtapp01/HousekeepingTasks.html

                                                            ii.      Click on the Add User To Portal button.

                                                         iii.      Use the Narrow Requesters Listing text box to enter a partial name of the user you want to add.

                                                          iv.      This string of characters will narrow down the list of users in the Select Requester E-Mail Address drop down listing.

                                                            v.      Once you have selected the associate you want to add to Admin Portal access, click on the Submit button to send them the request.

                                                          vi.      Notify the user that an E-Mail is on the way for them to register for the UDM Web Site.

                                                       vii.      The user need to open that E-Mail and follow the easy to use Registration instructions.

                                                     viii.      Once done, go to the User Attributes button (Just under the Add User To Portal button).

                                                          ix.      In the Modify Associate Attribute Settings Page, make sure the Check Boxes are checked for those Applications you want them to see.

c.      Removing user access from the Admin Portal

                                                              i.      Using your web browser, navigate to http://idmgmtapp01/HousekeepingTasks.html

                                                            ii.      Click on User Attributes button.

                                                         iii.      Find the user you wish to remove

                                                          iv.      Click on the Delete User radio button associated with the user you want to remove.

                                                            v.      A confirmation box will appear on the screen asking if you are sure you want to remove this user.

                                                          vi.      Click Yes to remove the user or Cancel to cancel the operation.

                                                       vii.      Once the user is removed, they will need to re-register if they need Admin Portal access back.

d.     Revoking access from the Admin Portal

                                                              i.      There may be situations where you do not want to remove a user, but you want to suspend their access to the Admin Portal.

                                                            ii.      To accomplish this, use your browse to the navigate to http://idmgmtapp01/HousekeepingTasks.html.

                                                         iii.      Click on User Attributes button.

                                                          iv.      Find the user you wish to revoke access to.

                                                            v.      Uncheck the Authorized check box for that user.

                                                          vi.      Once the box in unchecked, their access is automatically revoked.

                                                       vii.      To re-enable their access, simply re-check this box.

e.     Fixing Admin Portal Access

                                                              i.      Users who have access to the Admin Portal are said to have been registered.

                                                            ii.      When we say a user is Registered, that means an entry with their Employee ID is present in the WebNewUsers SQL table.

                                                         iii.      Each user who is granted access to the Admin Portal has two special entries made within their browser called Cookies.

                                                          iv.      These two Cookie entries are:

1.     user: This is their employee ID (i.e. 103100).

2.     ProdEncryptedKey : This is for security purposes and is created by the 'CreateEncryptedKey.exe' PowerShell script.

                                                            v.      These Cookie values provide both identification of the user to the website as well as an authentication encrypted code.

                                                          vi.      Without this encrypted code correctly written to the browser, the user will not be allowed access to the Admin Portal.

                                                       vii.      There may be instances when the balance between the users Cookies values and the information registered about the user get out of sync.

                                                     viii.      There are four situation that can cause this balance between the users laptop and the Admin Portal registration system to go Out-Of-Sync.

                                                          ix.      Those four categories are listed in more detail in the following paragraph, but for a quick overview, here they are:

1.     The user and Encrypted cookies do not exist on the user's laptop, but they are registered.

2.     The user and Encrypted cookies do not exist on the user's laptop and they are not registered.

3.     The user and Encrypted cookies do exist on the user's laptop and they are already successfully registered.

4.     The user and Encrypted cookies do exist on the users laptop, but they are NOT currently registered.

                                                            x.      Now we will discuss these four categories in detail:

1.     The user and Encrypted cookies do not exist on the user's laptop, but they are registered.

a.     This normally happens when a user decides to clear cache and 'Clear Cookies' is checked as an option.

b.     It can also happen if the user has more than one Browser profile set up and they use the second one for website access.

c.      There is a link the user can go to that can re-write the correct cookies values to their laptop.

d.     That link is: http://iuatidmgmtapp01/ResetWebsiteClient.html. They need to click on the option 'Recreate IDs'.

e.     The user needs to contact an Admin of the IDM website and that Admin will give them a code to reset their account.

f.       When accessing the website page, the user will need to entry both their employee ID and the code into the prompted areas.

g.     After this process is completed, they should be good to access the Admin Portal again.

2.     The user and Encrypted cookies do not exist on the user's laptop and they are not registered.

a.     In this case the user is a brand new associate to the IDM website.

b.     With this situation, we go through the normal new user registration process.

3.     The user and Encrypted cookies do exist on the user's laptop and they are already successfully registered.

a.     This is a case where a user was selected from the drop-down box that is already successfully registered.

b.     In this case they do not need to be registered again.

c.      A message is sent to the Admin stating they are registered and no action is necessary.

4.     The user and Encrypted cookies do exist on the users laptop, but they are NOT currently registered.

a.     This means they were once upon a time registered, but are not now because:

                                                                                                                                      i.      They were deleted from the Admin Portal registration tables using the delete user menu.

1.     This should never be the case because the delete user menu option also removes the user's cookies.

                                                                                                                                    ii.      Their entries were manually deleted from the Admin Portal registration tables using manual SQL commands. (Why!!).

b.     In this case, they need to contact the IDM team to re-register.
